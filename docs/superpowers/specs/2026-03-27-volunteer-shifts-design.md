# Volunteer Shifts ŌĆö Design Spec

**Date:** 2026-03-27
**Project:** JamHouse barrio coordination portal
**Feature:** Volunteer shift scheduling, sign-up, and knowledge base

---

## Overview

A role-aware volunteer shift management system accessible at `/admin/shifts`. Approved members can view the schedule, request open slots, and bail on shifts they've committed to. Admins can confirm assignments, manage shift definitions, configure schedule slots, and maintain a per-shift knowledge base.

---

## Identity & trust model

This app has no authenticated user identity ŌĆö no login, no session token, just a shared password stored in `sessionStorage`. There is no way to technically verify who is currently using the app.

For member self-service actions (request, cancel, bail), the member selects their own name from a dropdown populated from the approved members list (`Sheet1`). The UI trusts this selection. This is intentional: JamHouse is a small, trust-based group of friends, and social accountability is sufficient. Any approved member with the read password could technically submit an action on behalf of another name ŌĆö this is an accepted risk and not a bug.

`MemberName` in `ShiftAssignments` stores the selected name. The UI filters "your" assignments by comparing the selected name against `assignment.MemberName` ŌĆö members set their name once (persisted in `sessionStorage` as `jh_member_name`) and the UI uses it to highlight and enable self-service controls on their own chips.

## Data Model

Four new tabs in the existing Members Google Sheet (`SHEET_ID`).

### Tab: `ShiftDefs`

One row per shift type. Shift types can be added, edited, or removed by admins at any time.

| Column | Type | Description |
|---|---|---|
| `ShiftID` | string | Unique slug, e.g. `dinner`, `lnt-afternoon` |
| `Name` | string | Display name, e.g. "Dinner Shift" |
| `Description` | string | What the shift involves |
| `TimeExpectation` | string | e.g. "~2 hours after dinner" |
| `KBText` | string | Free-form knowledge base text (markdown) |
| `KBLinks` | string | Newline-separated `Label\|URL` pairs, one per line, e.g. `Meal Plan\|https://...` |

`KBLinks` example (multiple links):
```
Meal Plan|https://docs.google.com/...
Shopping List|https://docs.google.com/...
Kitchen Inventory|https://...
```

### Tab: `ShiftConfig`

One row per **slot** (a column in the schedule grid). Slots can be full days or specific time windows ŌĆö mixed freely.

| Column | Type | Description |
|---|---|---|
| `SlotID` | string | Unique slug, e.g. `tue-1`, `wed-am` |
| `Label` | string | Display label, e.g. "Tuesday 1 July" or "Wed 2 July 10:00ŌĆō12:00" |
| `Date` | string | ISO date, e.g. `2026-07-01` |
| `StartTime` | string | Optional, e.g. `10:00` |
| `EndTime` | string | Optional, e.g. `12:00` |

### Tab: `ShiftAssignments`

One row per assignment (a member assigned to a shift on a specific slot).

| Column | Type | Description |
|---|---|---|
| `AssignmentID` | string | Unique ID generated server-side: `Date.now() + '-' + Math.random().toString(36).slice(2,7)` |
| `SlotID` | string | References `ShiftConfig.SlotID` |
| `ShiftID` | string | References `ShiftDefs.ShiftID` |
| `MemberName` | string | Selected from approved members dropdown |
| `Status` | string | `requested`, `confirmed`, or `bailed` |

`AssignmentID` is generated by the API on `request` action ŌĆö never by the client.

### Tab: `MemberMeta`

One row per approved member. Stores per-member metadata that lives outside the main application data.

| Column | Type | Description |
|---|---|---|
| `MemberName` | string | Playa name or real name (matches `Sheet1` values) |
| `OtherResponsibilities` | string | Free-text, e.g. "Kitchen lead, build" ŌĆö editable by admins |

This tab is created and populated manually by admins (one row per approved member). The `shifts.js` endpoint fetches it and includes it in the response as `memberMeta`.

## Status Flow

```
(empty slot)
     Ōåō
 requested      ŌåÉ member signs up for an open slot
     Ōåō
 confirmed      ŌåÉ admin confirms the assignment
     Ōåō
   bailed       ŌåÉ member backs out of a confirmed shift
```

- **`requested`** ŌĆö member signed up, pending admin review
- **`confirmed`** ŌĆö admin has confirmed the assignment
- **`bailed`** ŌĆö member backed out after confirmation; slot flagged as needing coverage; row is kept (not deleted) so admins can see it
- Cancelled requests (member withdraws before confirmation) are deleted silently ŌĆö no record needed

## API Endpoints

All endpoints are POST with JSON body, consistent with the existing API pattern.

### `api/shifts.js` ŌĆö Read auth required

Fetches everything needed to render the page:
- All rows from `ShiftDefs`
- All rows from `ShiftConfig`
- All rows from `ShiftAssignments`
- All rows from `MemberMeta`

If any tab does not exist or is empty, return an empty array for that dataset ŌĆö never a 500.

**Request body:** `{ password }`
**Response:** `{ shifts, slots, assignments, memberMeta }`

### `api/shifts-request.js` ŌĆö Read auth required (member self-service)

Member requests, cancels, or bails on a shift slot. Read password is used ŌĆö the same password members use for budget/demographics. This is intentional: members do not have write passwords. The social trust model (see Identity section above) is the accepted security boundary.

**Request body:** `{ password, action, shiftId, slotId, memberName, assignmentId }`

- `action: "request"` ŌĆö creates a new `requested` assignment; `assignmentId` not required (generated server-side)
- `action: "cancel"` ŌĆö deletes a `requested` assignment; `assignmentId` required to identify the exact row
- `action: "bail"` ŌĆö updates a `confirmed` assignment to `bailed`; `assignmentId` required to identify the exact row

The client must pass `assignmentId` (from the loaded assignment data) for `cancel` and `bail` actions. The API looks up the row by `AssignmentID` column value, not by position.

### `api/shifts-update.js` ŌĆö Write auth required

Admin operations: structural changes and assignment management.

**Request body:** `{ password, action, ...payload }`

Supported actions:
- `confirm-assignment` ŌĆö set status to `confirmed`; payload: `{ assignmentId }`
- `remove-assignment` ŌĆö delete any assignment row; payload: `{ assignmentId }`
- `upsert-shift` ŌĆö create or update a `ShiftDefs` row; payload: shift object
- `delete-shift` ŌĆö remove a `ShiftDefs` row; payload: `{ shiftId }`
- `upsert-slot` ŌĆö create or update a `ShiftConfig` row; payload: slot object
- `delete-slot` ŌĆö remove a `ShiftConfig` row; payload: `{ slotId }`
- `update-member-meta` ŌĆö update `OtherResponsibilities` for a member; payload: `{ memberName, otherResponsibilities }`

## Page Structure

**Route:** `/admin/shifts`
**Files:** `admin/shifts.html`, `assets/js/admin-shifts.js`

### Member name selection

On page load, if `jh_member_name` is not set in `sessionStorage`, show a one-time name selector modal: a dropdown populated from the approved members list. The selected name is stored in `sessionStorage` as `jh_member_name` and used for all self-service actions. Members can change their name via a small "change" link near their name display.

### Auth levels

| Feature | Read auth (members) | Write auth (admins) |
|---|---|---|
| View schedule grid | yes | yes |
| View knowledge base | yes | yes |
| View summary panel | yes | yes |
| Request an open slot | yes | yes |
| Cancel own pending request | yes | yes |
| Bail on own confirmed shift | yes | yes |
| Confirm/remove any assignment | no | yes |
| Edit shift definitions | no | yes |
| Edit knowledge base | no | yes |
| Configure slots (dates/times) | no | yes |
| Edit member "other responsibilities" | no | yes |

### Layout ŌĆö three panels

**1. Schedule grid (main panel)**

- Rows = shift types (from `ShiftDefs`)
- Columns = slots (from `ShiftConfig`, sorted by date then `StartTime`)
- Each cell shows assigned members as name chips, colour-coded by status:
  - `confirmed` ŌåÆ solid chip
  - `requested` ŌåÆ muted/outlined chip
  - `bailed` ŌåÆ strikethrough chip with warning colour
- Members can click an empty cell to request that slot (submits with their `jh_member_name`)
- Members can click their own chip to cancel (if `requested`) or bail (if `confirmed`); "own" is determined by comparing chip's `MemberName` to `jh_member_name`
- Admins can click any chip for a context menu: confirm, remove
- Admins see an "+ Add" control in each cell
- Assignments whose `ShiftID` or `SlotID` no longer resolves (orphaned) are filtered out client-side and not rendered

**2. Knowledge base slide-out panel**

Triggered by clicking a shift name in the grid.

- Displays: shift name, description, time expectation, KB text (rendered as markdown), KB links as a clickable list (parsed from newline-separated `Label|URL` format)
- Members: read-only
- Admins: inline editing with save button

**3. Summary panel (right sidebar)**

Matches the layout of the existing spreadsheet's right panel.

- Lists all members (from `MemberMeta`) with: confirmed shift count (computed client-side from `ShiftAssignments`) and "other responsibilities" (from `MemberMeta.OtherResponsibilities`)
- Members highlighted if they have 0 confirmed shifts or have any `bailed` assignment
- Members: read-only
- Admins: can edit "other responsibilities" inline (calls `update-member-meta` action)

### Admin-only controls (shown only with write auth)

- **"Manage Shifts"** button ŌåÆ modal to add/edit/delete shift types
- **"Configure Slots"** button ŌåÆ modal to add/edit/delete schedule slots (date, optional start/end time, label)

## Error handling & edge cases

- Multiple members CAN share the same shift+slot (the grid shows chips per cell, plural)
- Member tries to request a slot they are already signed up for (non-bailed) ŌåÆ API returns 409, UI shows "already signed up"
- Member bails ŌåÆ `bailed` row persists, slot shown as open again for reassignment
- Shift deleted while assignments exist ŌåÆ orphaned assignments are filtered out client-side (ShiftID no longer resolves); admin should clean up via sheet directly if needed
- Slot deleted while assignments exist ŌåÆ same client-side orphan filtering
- Concurrent requests for the same slot ŌåÆ last write wins (acceptable for this scale)
- `ShiftDefs`, `ShiftConfig`, `ShiftAssignments`, or `MemberMeta` tab missing or empty ŌåÆ `shifts.js` returns empty array for that dataset, page renders with empty state
- `cancel` or `bail` sent without `assignmentId` ŌåÆ API returns 400
